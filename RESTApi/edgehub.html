POST /edgehub/api/connection

Enable or Disable the device connection.

  • Request Headers

    Log in with the OS or the "admin" account to get a JWT token (token field from the response body).

    Authorization: Bearer <token>

  • Request Body

    If the op is "connect" and the conncfg has been provided, it will set the connection setting first, then enable the connection to the cloud; if the conncfg is not provided, it will enable the connection only.

POST /edgehub/api/initialcode

Retrieve the current initial code for device registration.

  • Request Headers

    To access this endpoint, log in with an OS or "admin" account to obtain a JWT token (token field in the response body). Include the token in the request header.

    Authorization: Bearer <token>

  • Response 200 OK

    {
    "code": "9c46e07d81d247614cfb053bca94c45080b52086d425728aa88ba338b1cdd453"
}
    • code: The current initial code for the device registration process, valid for a specific time period before regeneration.

Initial Code Retrieval

The initial code required for device registration can be retrieved by either of the following methods:

  1. mDNS Browsing:

    • The initial code is broadcasted over the local network via mDNS under the service name _edgehub._tcp.local..
    • To retrieve the initial code, clients can browse the mDNS service and filter the TXT record, where the code field appears as code=<initial code>.
    • This method is suitable for trusted internal networks where mDNS broadcasts are accessible, and it allows clients to dynamically discover the initial code without direct API access.

  2. REST API Request:

    • Alternatively, clients can retrieve the initial code using the POST /edgehub/api/initialcode endpoint, which requires authorization via a JWT token obtained through login.
    • This approach provides direct access to the latest initial code when mDNS may not be available or accessible.

These options allow clients flexibility in retrieving the initial code based on network availability and security requirements.

POST /edgehub/api/device/challenge

Initiate a device challenge by generating a key that authenticates the request.

  • Request Headers

    This request does not require JWT-based authorization, as it relies on a challenge-response authentication mechanism.

  • Request Body

    {
    "clientID": "<string>",  // Unique identifier generated by the client
    "key": "<string>"        // HMAC and bcrypt-derived hash
}
    • clientID: A unique identifier generated by the client. This value is combined with an initial code (server-generated and updated periodically) to create a key.
    • key: A hashed value used to verify the client. The key is generated by combining clientID and initial code using HMAC and bcrypt.

  • Key Generation Process

    To generate the key, the client combines its unique clientID and the server-provided initial code (which can be obtained via mDNS or by calling POST /edgehub/api/initialcode). Here's the process in detail:

    1. Concatenate clientID and initial code into a single string.
    2. Generate an HMAC using SHA-256, with the initial code as the secret key and the concatenated string as the data.
    3. Hash the HMAC result with SHA-256 to ensure no NULL bytes are included, producing a consistent output for bcrypt.
    4. Hash the SHA-256 result using bcrypt to produce a secure, one-way key value.
    5. Convert the bcrypt result to a hex string to get the final key.

    Note: This approach ensures the key is unique and valid only while the initial code is active, providing secure and time-limited access to device registration.

    Example in Golang:

    func GenerateKey(initialCode, clientID string) (string, error) {
  data := fmt.Sprintf("%s%s", clientID, initialCode)
  h := hmac.New(sha256.New, []byte(initialCode))
  h.Write([]byte(data))
  hmacResult := h.Sum(nil)

  // Hash the HMAC result with SHA-256 to ensure no NULL bytes are included.
  sha256Result := sha256.Sum256(hmacResult)

  // Generate bcrypt hash from SHA-256 result.
  bcryptHash, err := bcrypt.GenerateFromPassword(sha256Result[:], bcrypt.DefaultCost)

  if err != nil {
    return "", err
  }

  return hex.EncodeToString(bcryptHash), nil
}

  • Response 200 OK

    {
    "code": "<string>"  // A new code for the client to use in subsequent requests
}

POST /edgehub/api/device/register

Complete the device registration process with clientID, code, and additional registration details. The device will connect to the EdgeHub once the registration process is successful.

  • Request Headers

    No JWT-based authorization is required for this endpoint, as it uses the challenge-response mechanism for validation.

  • Request Body

    {
    "clientID": "UKmEjSdv6Apcz4fC",
    "code": "d38c58c86d1dc04a0723940cb767454c45da3f54877d92c5b91aec7420417657",
    "timestamp": 1730158176,

    "baseUrl": "https://api-dm-center-master-ensaas.edge365.advantech.com/v1.1",
    "orgId": 10,
    "token": "eyJhbGciOiJIUzUxMiIsInR5cCI6IkpXVCJ9..."
}
    • clientID: Unique identifier generated by the client.
    • code: Generated response code based on the challenge code and timestamp provided during the challenge-response process.
    • timestamp: The current client-side time when generating code.

    Additional Registration Fields:

    • baseUrl: The base URL for the device management API, selected from the station list.

      • https://bi.wise-deviceon.advantech.com/api/stationList
      • https://wise-deviceon-bi.azurewebsites.net/api/stationList

      • Within each station's data, the url_apidm field represents the possible baseUrl values. For example:

        {
    "stationList": [
        {
            "name": "EdgeSync360/EdgeHub Master Station",
            "url": "https://pivot-api-hub-master-ensaas.edge365.advantech.com",
            "url_apidm": "https://api-dm-center-master-ensaas.edge365.advantech.com/v1.1"
        }
    ]
}

    • orgId: Organization ID from the selected tenant's organization data. This value is retrieved from the tenant array in the {baseUrl}/connect/login response.

    • token: JWT access token obtained from the {baseUrl}/connect/login response. The accessToken field in the response body represents this token.

  • Code Generation Process

    The client generates code as follows:

    1. Concatenate clientID, challenge code, and timestamp into a single string.
    2. Generate an HMAC using SHA-256 with the initial code as the secret key and the concatenated string as data.
    3. Convert the HMAC result to a hex string to produce the final code.

    Example in Golang:

    func GenerateResponseCode(initialCode, clientID, challengeCode string, timestamp int64) string {
    data := fmt.Sprintf("%s%s%d", clientID, challengeCode, timestamp)
    h := hmac.New(sha256.New, []byte(initialCode))
    h.Write([]byte(data))
    return hex.EncodeToString(h.Sum(nil))
}
